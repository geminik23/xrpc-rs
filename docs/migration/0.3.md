# Migrating to xrpc-rs 0.3

xrpc-rs 0.3 makes client and transport shutdown deterministic. The main source changes are explicit client startup, optional receive-task joining, wildcard error matching, and stricter transport lifecycle contracts.

The minimum supported Rust version is 1.85.

## Start, close, and optional join

Prefer fallible startup. Retain the returned handle when strict receive-task completion confirmation or forced shutdown is required:

```rust
let client = RpcClient::new(channel);
let handle = client.try_start()?;

let response: Response = client.call("method", &request).await?;

let close_result = client.close().await;
let join_result = handle.join().await;
close_result?;
join_result?;
```

`try_start()` succeeds exactly once. Calling an RPC method before startup returns `RpcError::ClientError`; work admitted after terminal close returns `RpcError::ConnectionClosed`.

Keeping the returned handle is not required for ordinary RPC calls. Dropping it detaches public join ownership but does not stop the receive loop. `RpcClient::close()` still performs the graceful close workflow.

`RpcClient::close()` is graceful and single-flight. It marks the client terminal immediately, then runs the complete close workflow in a detached task before waiting for in-flight admission. It cancels blocked sends that have not committed, but waits for committed unary and raw calls to receive a response or reach their configured timeout. Cancelling the first close caller does not abandon shutdown, even if it was still waiting for admission. This prevents a non-idempotent RPC that was already sent from being reported as closed locally.

`RpcClientHandle::join()` waits without aborting. Use `RpcClientHandle::shutdown()` only as a forced fallback. Dropping a client performs best-effort cleanup. Explicit `close()` is the normal shutdown path; following it with `join()` additionally confirms receive-task termination when the handle was retained.

## Error matching

`RpcError` and `TransportError` are now `#[non_exhaustive]`. External exhaustive matches need a fallback arm:

```rust
match error {
    RpcError::ConnectionClosed => reconnect_or_stop(),
    RpcError::Transport(TransportError::Timeout { .. }) => retry(),
    other => report(other),
}
```

`TransportError::ConnectionClosed` represents an explicit local close or a clean peer close at a frame boundary. `NotConnected` remains for a connection that is absent but not known to be terminal. A timeout remains non-terminal only when no frame bytes or SHM commit state were consumed.

## Load-balanced clients

Call `LoadBalancedClient::close()` to close and join every managed child client. Child closes are initiated concurrently, and concurrent or cancelled close callers observe the stored result.

Failover now retries only failures that occur before `RpcClient::call()` is invoked, such as endpoint selection, connection creation, or client startup failure. Once an RPC is invoked, xrpc-rs does not retry it automatically because the remote commit state is unknown and retrying could duplicate a non-idempotent operation. `failover_enabled = false` limits setup to one server attempt.

Cached clients are validated against the current endpoint rather than only the mutable server index. Active request counts and stream affinity are released automatically when their futures or receivers are dropped.

## Shared-memory mappings

The 0.3 shared-memory layout uses protocol version 2 with explicit readiness and layout validation. It is intentionally incompatible with older mappings. Stop all old processes and recreate the named SHM regions during upgrade.

A named SHM server supports one attached client. A second client receives an error because the directional buffers are SPSC queues. Use distinct names for additional client/server pairs.

The maximum payload is `buffer_size - 5` bytes because the ring stores a four-byte length prefix and reserves one sentinel byte.

## Socket framing

TCP and Unix transports permit concurrent reads and writes. A clean EOF before a new frame is `ConnectionClosed`; EOF, timeout, or cancellation after consuming or publishing part of a frame makes the connection terminal because the byte stream cannot be safely re-framed.

`UnixFrameTransport::from_stream()` and `TcpFrameTransport::from_stream()` are fallible because cloning the socket handle required for full-duplex shutdown can fail. Handle their `TransportResult` return values.

Outbound socket payloads cannot exceed `u32::MAX` even if a larger `max_message_size` is configured.

## Message framing

The 0.3 encoder writes `Msg Length` as the exact byte count after the 10-byte
magic/version/flags/length prefix. It also counts all three variable-field length
words in the 16 MiB limit. The decoder validates the declared length and every
variable field before reading it, so truncated or oversized input returns a
protocol error instead of panicking.

The decoder remains compatible with 0.2.x frames whose declared message length
was exactly 10 bytes too small. New frames always use the corrected length, which
older Rust decoders also accept because they did not require an exact match.

Servers created with a stateful custom codec now use that configured codec
instance for unary and streaming error responses as well as successful replies.
